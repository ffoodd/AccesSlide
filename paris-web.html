Démythifions

La contre-performance des sélecteurs

La rumeur court depuis plusieurs années que les sélecteurs universels, d'attributs ou d'adjacence ne seraient pas performants — et sont par conséquent à éviter.

C'est faux. Et c'est prouvé par Ben Frain en 2014.

La différence  — qui, certes, existe — est absolument négligeable. En revanche, soyez prudents avec les propriétés que vous utilisez.

Démythifions encore

La séparation du fond et de la forme

On s'en sert à tort et à travers, de celle-là. Généralement contre une idée et sans réfléchir — comme un axiome. Pourtant dans le web, cet univers en HTML et CSS, sa seule portée pratique pertinente est de ne pas utiliser d'attributs de présentation dans le HTML ni de balises sémantiques pour leur aspect visuel. Saÿtou.

Et si on prône ça, c'est juste parce que sinon le fond devient inintelligible.

Pour tous les autres cas, invoquer la séparation du fond et de la forme est inepte, pour une raison pourtant simple : la forme dépend du fond. Ça n'est guère plus compliqué que ça.

J'en ai parlé plus en détail dans « Le sens de la sémantique ».

Les sélecteurs pestiférés

La mode en ce moment — fidèlement aux mythes précédemment évoqués — est de n'utiliser que des classes. Des classes partout ! Saÿcon. En plus, c'est moins coloré syntaxiquement dans mon éditeur.

Voyons nos vilains petits canards :

  1. Le sélecteur universel *
  2. Les sélecteurs d'attributs complexes [attr*=""]
  3. Les sélecteurs d'adjacence et de parenté * + *, * > *, * ~ *
  4. Les sélecteurs d'élément surqualifiés html[class*=""]

Moi, je les aime beaucoup. Ils sont rigolos. Allez, jetons-y un œil.

Les micro-données

Les aficionados du SEO les connaissent bien : elles ont vocation à enrichir la sémantique d'un document pour compléter la sémantique HTML.

La tendance actuelle est plutôt le format JSON-LD dans une balise script, mais je favorise HTML.

            
              [itemprop="author"] {
                  font-variant: small-caps;
              }

ce sélecteur présente les mêmes avantages que si vous employiez une classe .author (poids du sélecteur, indépendance vis-à-vis de la balise), mais vous y ajoutez une dimension sémantique.

ARIA

ARIA est un ensemble d'attributs qui permet de rendre accessibles certains motifs de conception et les interactions qui y sont associées.

C'est la version bio des plugins jQuery de votre enfance.

            
              [role="tabpanel"][aria-hidden="true"] {
                  display: none;
              }

Et voilà ! Pourquoi utiliser des classes — que vous manipulez en JS de toute façon — quand vous disposez d'une panoplie complète d'attributs ?

L'économique de répétition

Les sélecteurs d'attributs peuvent fonctionner comme des expressions régulières. Et ça, c'est vachement pratique. On peut s'en servir pour mettre en place des motifs, ce que vous faites déjà avec des méthodologies comme BEM, SMACSS ou OOCSS. Et si comme moi, vous trouvez que quelquechose cloche dans le class="btn btn-lg btn-default" de Bootstrap, vous devriez apprécier ce qui suit : 

              
                [class*="btn"] {
                  border-width: .25em;
                }

                [class*="primary"] {
                  background-color: #032c36;
                  color: #efeee3;
                }

                [class*="-xs"] {
                  font-size: smaller;
                }

Placer dans le HTML la classe btn-primary-xs permettra d'appliquer les trois déclarations ! J'aime beaucoup car ça permet de s'appliquer à nommer et abstraire son barda pour en faire des motifs de conception viables.

Gaffe aux téléscopages : tout ce qui aura une classe comprenant -xs sera plus petit. Logique, mais ça peut surprendre malgré tout :)

La chouette lobotomisée

Crédit à Heydon Pickering sur AListApart.

Ces smileys vont sûrement vous faire crier, mais honnêtement pourquoi galérer à gérer vos espacements sur des classes, quand ce sélecteur fait le job pour vous ?

            
              * + * {
                  margin-top: 1.5rem;
              }

La chouette dubitative et la chouette duck face

Le sélecteur d'adjacence directe est pratique, mais celui d'adjacence indirecte également (* ~ *) — et celui de parenté (* > *), je vous raconte même pas ! Utile lorsque l'affichage d'un enfant direct dépend de celui d'un prédécesseur, par ex. : 

            
              main > *:target ~ * {
                  display: none;
              }

Devinez quoi ? Cet exemple aussi est d'Heydon Pickering, sur SmashingMagazine cette fois.

Le garde-fous

Progressons dans l'hérésie : et si nous ajoutions futilement du poids à nos sélecteurs, pour rigoler ?

Saÿmal, pas vrai ? Pourtant tous les frameworks CSS devraient en abuser, notamment parce qu'ils pourraient ainsi contribuer à améliorer la qualité globale des sites produits grâce à eux.

L'astuce du box-sizing sur html

Vous la trouvez par exemple dans Foundation et dans Bootstrap V4. Imaginez un peu si ces frameworks ne copiaient pas bêtement cette astuce, mais y ajoutaient un seul, tout petit attribut [lang]. 

            
              html[lang] {
                box-sizing: border-box;
              }

              *, *::after, *::before {
                box-sizing: inherit;
              }

ÀMHA, les utilisateurs auraient tôt fait de préciser la langue du document, et une des erreurs d'accessibilité les plus courantes serait tout à coup menacée d'extinction !

L'autre garde-fous

Nous savons tous que les images sont un point particulièrement sensible du web. Est-ce qu'elle véhicule de l'information ? Comment doit-elle s'afficher ? Bla bla bla.

Par principe, et ce depuis l'avènement du RWD, on colle un petit bout de CSS par défaut sur img :

            
              img {
                  max-width: 100%;
              }

On le trouve dans Knacss, dans Röcssti et, certes un peu mieux caché, dans Bootstrap.

Ma suggestion

            
              img[alt] {
                  max-width: 100%;
              }

Là aussi, ça devrait quelque peu altérer le web devant vos yeux. Et croyez-moi, plus personne dans votre équipe n'omettra une alternative. :)

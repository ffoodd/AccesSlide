Définitions

🤖 Test automatisé
Test dirigé par un automate.
Note : les bookmarklets et extensions navigateurs relèvent du test manuel.
🎯 Front-end
Un document HTML, disponible via un chemin, éventuellement saupoudré de CSS et JS.
🗺 Plan de test
Stratégie adoptée pour déterminer quels sont les tests à effectuer, la qualité ciblée, les intervenants, les outils, le cycle de vie, les livrables, le référentiel… en fonction d’un objectif précis.
📈 Budget
Le budget est primordial afin de préciser un périmètre à cibler — et ne pas devenir un nazi de la qualité.

Les bons plans

Le bon plan ?
C’est un plan, il a un test, il voit une erreur, il crash…

Source inconnue

Deux axes pour un bon plan :

  1. Les résultats doivent être intelligibles ;
  2. un référent doit être identifiable.

Sans ça, vous pissez dans un violon 🎻.

Le cambouis

Par l’axe du quoi

Pour chaque domaine, je proposerai mon outil favori et son mode d’emploi, et mentionnerai quelques alternatives. Précisions :

  • Où tester : dans le navigateur (via extensions ou bookmarklets), en local dans l’IDE ou avec un navigateur décapité, dans les tâches automatisées de construction du code, dans une machine virtuelle ou un conteneur, etc.
  • Quand tester : dès la modification du code, à chaque commit, à chaque fusion de branche, à chaque déploiement, perpétuellement en production…
  • Avec quoi tester : un outil par domaine — en changeant le barème ?, utiliser un outil clé-en-main en SaaS, ou encore écrire (et maintenir) nous-mêmes les tests…
  • Comment gérer les résultats : générer des flux de données en JSON, des rapports PDF prêt-à-mailer, des fichiers markdown historisés, créer des tickets à la volée (bonjour le spam), lever une alerte ou bloquer un déploiement en cas d’erreur critique, etc.
